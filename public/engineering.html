01 · Três caminhos para construir

Antes de escrever uma linha,
escolha como vai construir.

A decisão sobre quem escreve o código define orçamento, velocidade e risco. Os três caminhos têm trade-offs reais.

Caminho A

Time interno

1 tech lead + 2 devs + 1 designer. Contratação direta ou PJ.
Custo/mêsR$ 65k – 110k
MVP em4–6 meses
Riscomédio
Quando faz sentido: você levantou capital (R$ 1M+) e tem 12+ meses de runway. Quer time que vira o produto, não freelancers.
Caminho B · recomendado

CTO co-founder + IA

Um sócio técnico equity, com pair-programming via Claude Code / Cursor para acelerar.
Custo/mêsR$ 3k – 8k
MVP em3–4 meses
Riscobaixo
Quando faz sentido: bootstrap. Você precisa validar antes de levantar. Equity (20–35%) por código + decisões técnicas. Um CTO certo é o multiplicador de tudo.
Caminho C

Agência / no-code

Bubble, Webflow + Xano, ou agência fechada. Você não toca código.
CustoR$ 60k – 200k once
MVP em2–3 meses
Riscoalto
Quando faz sentido: só para validar a tese rapidíssimo. Não escala para o que Stackly precisa (busca vetorial, AI, custom). Use só para landing + waitlist.
Veredito
Vá de Caminho B. Stackly precisa de IA de verdade, busca vetorial, embeddings, integrações com Stripe — coisas que não-code não resolve bem. E o ROI de um CTO sócio é assimétrico: 30% de equity para alguém que entrega o produto inteiro é o melhor deal possível. Use o tempo entre achar o CTO e ter o MVP para acelerar com Claude Code + esse blueprint.
02 · Princípios de engenharia

O que decidimos
de cara — para sempre.

01 · BORING TECH

Tecnologia chata vence.

Next.js + PostgreSQL + Stripe. Não vamos reinventar nada que não seja o core do negócio. Cada vez que alguém propor uma stack "moderna que vai resolver", a resposta é não. Inovação só onde dá vantagem competitiva: IA, recomendação, score.

02 · TYPE-SAFE END-TO-END

TypeScript em toda parte.

Frontend, backend, schema do DB. Drizzle ORM gera tipos do Postgres. Zod valida runtime + gera tipos. Bugs de "campo undefined" não existem aqui.

03 · SERVERLESS-FIRST

Não cuidamos de servidor.

Vercel cuida do app. Supabase cuida do banco e auth. Inngest cuida das jobs. Stripe cuida de pagamento. Em troca de uma margem maior de infra, ganhamos meses do CEO/CTO não ficando ajustando deploy.

04 · OBSERVABILITY DAY ZERO

Você não conserta o que não vê.

Sentry + Posthog + Axiom desde o primeiro deploy. Saber onde o produto falha é mais importante que esconder que falha.

05 · SEO É PRODUTO

SSR/SSG, não SPA.

Cada perfil de SaaS, cada ranking, cada comparativo é uma página renderizada no servidor. Google indexa tudo. Performance perfeita (Core Web Vitals verde). SEO programático é metade da estratégia de aquisição.

06 · IA COMO FERRAMENTA, NÃO DEUS

Function calling sempre.

O LLM nunca inventa um produto que não existe. Ele chama nossa busca, chama nosso comparador. Hallucinations matam credibilidade — function calling + RAG + validação evita isso.

03 · Stack tecnológica

Cada camada,
uma escolha defensável.

Sete camadas. Para cada uma, a escolha, o porquê em uma frase, e o que rejeitamos.

Camada
Frontend
SSR · SPA · SEO
Next.js 15 React 19 Tailwind v4 shadcn/ui

App Router para SSR + ISR + streaming. SEO programático nativo. Server Components reduzem JS no cliente. shadcn dá componentes prontos sem o lock-in de uma lib pesada — copia para o repo, edita.

Rejeitamos
Remix: ecossistema menor.
SPA puro (Vite): zero SEO.
Astro: app dinâmico demais.
Camada
Backend / API
routes · jobs
Next.js Route Handlers Drizzle ORM Zod Inngest

API mora no mesmo repo do front — sem coordenação entre dois projetos. Drizzle dá tipos do banco direto no código. Zod valida tudo na fronteira (input do user, output do LLM). Inngest pra jobs assíncronos (gerar embedding, recalcular score, enviar email).

Rejeitamos
NestJS: overhead p/ MVP.
Hono separado: dois deploys.
Prisma: lento, abstrai demais.
Camada
Banco de dados
core + auth + vetorial
Supabase PostgreSQL 16 pgvector RLS

Postgres faz tudo que precisamos: relational, full-text search (pg_tsvector), vetorial (pgvector), JSONB para metadata flexível. Supabase entrega Postgres + Auth + Storage + Realtime + Edge Functions em um único produto. Row Level Security para multi-tenant.

Rejeitamos
MongoDB: schema-less é prejuízo.
Firebase: vendor lock-in fatal.
Pinecone separado: pgvector é suficiente até 10M vetores.
Camada
IA / LLMs
consultor · score · gen
Claude Sonnet 4.5 Claude Haiku 4.5 OpenAI text-embedding-3 Vercel AI SDK

Sonnet para o consultor IA (tarefas longas, function calling robusto). Haiku para gerar resumos, tags automáticas, micro-tarefas (10× mais barato). OpenAI embeddings porque são o padrão e funcionam bem com pgvector. Vercel AI SDK abstrai streaming e function calling.

Rejeitamos
GPT-4 only: Claude é melhor em PT-BR.
Llama self-hosted: custo de infra mata.
LangChain: abstração demais p/ pouco ganho.
Camada
Pagamentos
assinatura · PIX
Stripe Asaas Stripe Connect

Stripe para tudo internacional, cartão, billing portal. Asaas ou Pagar.me para PIX recorrente (Stripe BR ainda não é completo). Stripe Connect para split de afiliados quando crescer. Trial sem cartão usa flag interno (não Stripe).

Rejeitamos
Hotmart/Eduzz: má UX/API.
Mercado Pago: API mais frágil.
Próprio: PCI compliance é inferno.
Camada
Comunicação
email · push · WA
Resend React Email Twilio WhatsApp Knock

Resend para email transacional e marketing — API limpa, deliverability boa. React Email para escrever templates como componentes. Twilio para notificações WhatsApp aos anunciantes (novo lead, review). Knock centraliza notification preferences quando crescer.

Rejeitamos
SendGrid: pesado e caro.
Mailgun: deliverability BR ruim.
SES: setup é projeto.
Camada
Observability
analytics · logs · erros
PostHog Sentry Axiom Vercel Analytics

PostHog para product analytics, session replay, feature flags, A/B test — tudo no mesmo lugar. Sentry para errors e performance. Axiom para logs estruturados. Vercel Analytics para Core Web Vitals.

Rejeitamos
GA4: ruim p/ produto.
Datadog: caro demais p/ MVP.
Mixpanel: PostHog ganha em features.

05 · Modelo de dados

Doze tabelas centrais.
O resto é detalhe.

Visualização do schema. Em seguida, o SQL das quatro tabelas que sustentam tudo.

A tabela products

É o coração do sistema. O JSONB screenshots evita ter uma tabela separada só pra imagens. O campo saas_score é cacheado aqui pra leitura O(1), mas recalculado por um Inngest job à noite. 

CREATE TABLE products (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id uuid REFERENCES organizations(id),
  slug text UNIQUE NOT NULL,
  name text NOT NULL,
  description text,
  screenshots jsonb DEFAULT '[]',
  status text DEFAULT 'draft',
  saas_score numeric(4,2),
  created_at timestamptz DEFAULT now()
);

A tabela product_vectors

Aqui mora a inteligência de busca. Cada produto é quebrado em chunks ("resumo", "features", "preço") e transformado em vetor de 1536 dimensões pela OpenAI. O índice HNSW permite busca KNN sub-milissegundo. 

-- Habilita extensão pgvector
CREATE EXTENSION IF NOT EXISTS vector;

CREATE TABLE product_vectors (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  product_id uuid REFERENCES products(id),
  content_chunk text NOT NULL,
  embedding vector(1536) NOT NULL
);

CREATE INDEX ON product_vectors
USING hnsw (embedding vector_cosine_ops);
06 · API Surface

Restrita e tipada.

Como SSR resolve 80% das leituras, a API é usada só para mutações e interações ricas (chat). O formato é padrão: Zod IN, JSON OUT.

Endpoint Role Input (Zod) Output
GET /api/products Public ?q=crm&cat=sales { "data": Product[] }
POST /api/products Org Admin z.object({ name: z.string(), ... }) { "id": "uuid" }
STREAM /api/chat User z.object({ messages: Message[], chatId: z.string() }) text/event-stream (Vercel AI SDK)
POST /api/ai/compare Public z.object({ p1: z.string(), p2: z.string() }) { "verdict": "string", "winner": "id" }
POST /api/webhooks/stripe Stripe System StripeEvent (validado via signature) 200 OK

Server Actions vs Route Handlers

Para submissão de formulários simples no painel do anunciante (ex: editar perfil), usamos React Server Actions. Para consumo por outros serviços (ex: Inngest) ou endpoints públicos, usamos Route Handlers (/api/*).

07 · Pipeline de IA

O cérebro do consultor.
Sem hallucinations.

O LLM não é um oráculo mágico. Ele é um motor de raciocínio que recebe ferramentas rígidas para operar sobre nosso banco de dados. Isso se chama RAG + Function Calling.

Input do Usuário
"Preciso de um CRM p/ clínica de estética, focado em WA"
STEP 1 Intent Router (Haiku)
TOOL CALL searchProducts args: { category: "CRM", query: "clínica estetica whatsapp" }
→ Busca no pgvector (HNSW) retorna Top 3 (Zendesk, RD Station, Exact)
STEP 2 Synthesis Planner (Sonnet 3.5)
Avalia o Top 3 retornado pela ferramenta de busca, compara os JSONs dos produtos, e escreve a resposta final recomendando o RD Station justificando a integração nativa com WA e o preço adequado à clínica.

Function Tool: searchProducts

A ferramenta que damos ao LLM. Quando ele decide chamá-la, a execução pausa, rodamos a query KNN no Postgres localmente, e devolvemos os resultados pro LLM. Ele não precisa "saber" sobre os produtos, só precisa saber pedir pra nossa tool.

System Prompt Base

You are Stackly's AI Consultant.
Your job is to recommend B2B SaaS.
RULES:
1. NEVER invent products or features.
2. ALWAYS use the `searchProducts` tool first.
3. If no product fits perfectly, admit it.
4. Keep tone professional but direct.
08 · Roadmap MVP

Da linha 1 ao
primeiro boleto pago.

Timeline cravada em 16 semanas. Focada no caminho "CTO Co-founder + Bootstrap".

Semanas 1–4
Mês 01 · Fundação & Catálogo Público

Setup do repo, banco, CI/CD. Criação das tabelas core. Importação de 200 SaaS seed via script. Frontend público (SSR): home, página de produto individual, categorias.
Marco: O site existe no ar, indexável pelo Google, e pode ser navegado (ainda sem login).

Semanas 5–8
Mês 02 · Painel do Anunciante & Auth

Supabase Auth ativado. Fluxo de onboarding do anunciante (reivindicar SaaS ou criar novo). Dashboard B2B básico (edição de perfil, métricas fake/iniciais).
Marco: Donos de SaaS podem criar conta e editar sua própria página.

Semanas 9–12
Mês 03 · Consultor IA & Embeddings

Job Inngest para gerar embeddings de todos os produtos ativos. Interface do chat pro usuário final. Integração com Vercel AI SDK e Claude. Function calling para o LLM buscar produtos e retornar cards renderizados na UI (generative UI).
Marco: A mágica principal do produto funciona. Usuário pergunta, IA responde com cards reais.

Semanas 13–16
Mês 04 · Monetização & Lançamento

Integração com Stripe Checkout. Bloqueio de features Pro para anunciantes (SaaS Score detalhado, leads). Refinamento de design, QA mobile. Envio de emails em lote para a waitlist de 1.200 pessoas.
Marco: MVP lançado. Primeiro R$ 1 gerado.

10 · Riscos & Mitigações

O que pode quebrar
essa arquitetura.

Risco de Dados Probabilidade: Média

Cold Start Database

Se lançarmos sem SaaS cadastrados, o marketplace é inútil. E o LLM não terá o que recomendar.

Mitigação: Antes da semana 4, criaremos um script em Node para fazer scrap amigável de dados públicos de 200 SaaS principais do Brasil e faremos o seeding no banco como "Drafts". O perfil existe, o dono só precisa "dar claim".

Risco de IA Probabilidade: Alta

Latência do Consultor IA

O ciclo (User → Edge → DB pgvector → LLM Provider → Edge → User) pode demorar 5-8 segundos. UX péssima para chat.

Mitigação: Obrigatório usar Streaming Server-Sent Events (SSE) com Vercel AI SDK. O usuário começa a ler as palavras 300ms após o clique, enquanto o LLM ainda está gerando o final da frase.

Risco de Vendor Probabilidade: Baixa

Lock-in do Supabase

Ficarmos presos na nuvem do Supabase se o preço deles subir muito.

Mitigação: Supabase é PostgreSQL open-source por baixo. Se precisarmos sair, exportamos o pg_dump e rodamos no AWS RDS. O custo de migração existe (refazer as Edge Functions e Auth), mas os dados não são reféns.
